INTEGRANTE 1 - ARQUITECTURA, AUTENTICACION Y NAVEGACION
=======================================================

PROPOSITO DE ESTAS GUIAS
------------------------
El proyecto se divide entre cinco expositores, pero todos deben comprender el flujo completo.
Primero se explica ese flujo general y luego la parte que esta persona debe desarrollar con
mayor profundidad. Las rutas indicadas parten de la raiz D:\MOVILES_PRO_2.

FLUJO GENERAL COMPLETO DE EXCHANGEPRO
-------------------------------------
1. Android inicia MainActivity. onCreate activa edge-to-edge, aplica ExchangeProTheme y llama
   a ExchangeProNavGraph.
2. El NavHost empieza en Login. LoginScreen valida correo/contrasena y llama a
   FirebaseAuthRepository.signIn.
3. Firebase Authentication valida las credenciales. Luego Firestore lee users/{uid} para
   conocer el rol. USER va a Home; ADMIN va a AdminDashboard.
4. Al registrar una cuenta, FirebaseAuthRepository.register crea la identidad en Authentication
   y un batch crea users/{uid}, wallets/{uid} y balances PEN, USD, EUR con saldo cero.
5. Home consulta perfil, wallet, ofertas y transacciones mediante repositorios. La UI nunca
   escribe directamente en Firebase: llama a la capa data/repository.
6. Wallet permite recargar o retirar. Las operaciones usan transacciones atomicas de Firestore:
   saldo, movimiento y recarga/retiro se guardan juntos o no se guarda nada.
7. El usuario registra Yape, Plin o banco en paymentData/{uid}. Estos destinos se usan para
   retiros y para informar medios de pago.
8. Al crear una oferta, FirebaseOfferRepository retiene saldo y crea offers/{offerId}. Al tomarla,
   FirebaseTransactionRepository reduce la oferta y crea transactions/{transactionId}.
9. Quien paga selecciona una imagen. ImageCompressor la reduce a JPEG <= 500 KB;
   FirebaseAttachmentRepository la guarda como Blob en attachments y cambia la transaccion
   de PENDIENTE_PAGO a PAGADO.
10. El propietario de los fondos verifica el comprobante y libera. complete mueve el retenido
    a la contraparte, crea movimientos y marca COMPLETADO.
11. Si existe un problema, una disputa congela la transaccion en EN_DISPUTA. El administrador
    resuelve si libera a la contraparte o devuelve al propietario.
12. Feedback, calificaciones y notificaciones tambien son documentos Firestore. Las reglas
    identifican propietario, participantes y rol ADMIN.

COMO LEER EL CODIGO
-------------------
La secuencia habitual es:
Pantalla Compose -> validacion local -> coroutine/callback -> Repository -> Firebase -> modelo
de dominio -> estado Compose -> recomposicion de pantalla.

Carpetas:
- presentation: pantallas, estados de UI y navegacion.
- data/repository: consultas y operaciones Firebase.
- data/image: compresion de imagen.
- domain/model: objetos compartidos y estados del negocio.
- ui: tema y componentes reutilizables.

EXPLICACION PASO A PASO
-----------------------
A. MainActivity
- Es el punto de entrada declarado en AndroidManifest.xml.
- setContent cambia el enfoque XML tradicional por Jetpack Compose.
- ExchangeProTheme aplica colores y tipografias.
- ExchangeProNavGraph crea el controlador de navegacion y todas las rutas.

B. LoginScreen
- Mantiene email, password, mensajes y estado loading con remember/mutableStateOf.
- Valida campos antes de llamar al repositorio.
- signIn no confia en un correo especial: autentica con Firebase y lee el rol del documento.
- popUpTo elimina Login del historial, evitando volver con el boton atras.

C. FirebaseAuthRepository.signIn
- signInWithEmailAndPassword autentica.
- Recupera uid desde FirebaseAuth.
- Lee users/{uid}; transforma "USER" o "ADMIN" a UserRole.
- Si falta perfil/rol, cierra sesion para no dejar una identidad incompleta.
- readableError convierte errores tecnicos en mensajes entendibles.

D. Registro
- RegisterScreen valida nombres, apellidos, correo, telefono de 9 digitos, documento de 8,
  contrasena y confirmacion.
- register crea primero el usuario de Authentication.
- Un WriteBatch crea perfil, wallet y balances. El rol siempre nace como USER.
- Si falla Firestore, elimina la identidad recien creada para evitar una cuenta huerfana.
- Un administrador se obtiene cambiando manualmente role a ADMIN en Firestore; nunca se
  codifica una clave administrativa dentro de la APK.

E. Navegacion
- Routes.kt centraliza cadenas; evita escribir rutas diferentes en cada pantalla.
- ExchangeProNavGraph registra Login/Register sin drawer y pantallas internas con scaffold.
- ExchangeScaffold define drawer, top bar y cerrar sesion.
- isAdmin elige menu usuario o menu administrador.

F. UI reutilizable
- ExchangeComponents.kt contiene ExchangeCard, PrimaryAction, SecondaryAction y StatusPill.
- Una modificacion alli afecta muchas pantallas; por eso reduce duplicacion.

PERMISOS, DEPENDENCIAS Y CONFIGURACION
-------------------------------------
- AndroidManifest.xml declara INTERNET. El selector de imagen del sistema evita permisos
  generales sobre la galeria.
- app/build.gradle.kts incluye Compose, Navigation, Firebase Auth, Firestore y Analytics.
- google-services.json vive en app/, conecta la APK con el proyecto Firebase y esta ignorado
  por Git.
- local.properties contiene la ruta del SDK local y tampoco debe subirse.
- minSdk/targetSdk/applicationId estan en app/build.gradle.kts.

DEMOSTRACION SUGERIDA
---------------------
1. Mostrar MainActivity y seguir la llamada hasta ExchangeProNavGraph.
2. Registrar un usuario y mostrar Authentication + users + wallet/balances en Firebase.
3. Iniciar como USER y luego como ADMIN para demostrar enrutamiento por rol.
4. Abrir drawer y relacionar cada opcion con Route.

IDEA CENTRAL PARA EXPLICAR
--------------------------
Authentication responde "quien es"; el documento users/{uid} responde "que rol y perfil
tiene"; Navigation decide "que interfaz debe ver".
