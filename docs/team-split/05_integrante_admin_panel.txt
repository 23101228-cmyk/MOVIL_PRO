INTEGRANTE 5 - ADMINISTRACION, SEGURIDAD, REPORTES Y CIERRE TECNICO
==================================================================

PROPOSITO DE ESTAS GUIAS
------------------------
Esta persona explica como el administrador observa y resuelve el sistema, y cierra la exposicion
con seguridad, configuracion y limitaciones academicas.

FLUJO GENERAL COMPLETO DE EXCHANGEPRO
-------------------------------------
1. MainActivity carga tema/NavGraph/Login.
2. Auth autentica y users/{uid}.role enruta USER o ADMIN.
3. Registro crea perfil/wallet/balances.
4. USER consulta Home, wallet, medios, ofertas y operaciones.
5. Recarga crea saldo/movimiento; oferta retiene; tomar crea transaccion.
6. Imagen comprimida confirma pago.
7. Liberacion completa y mueve fondos; disputa congela.
8. ADMIN ve metricas y pendientes.
9. ADMIN resuelve disputa atomicamente o responde feedback.
10. Reportes consultan colecciones reales.
11. Notificaciones comunican eventos.
12. firestore.rules limita operaciones.

COMO LEER EL CODIGO
-------------------
AdminScreens.kt es presentacion. FirebaseAdminRepository concentra consultas globales y acciones
privilegiadas. firestore.rules es la ultima barrera: la UI oculta acciones, pero las reglas
deciden si Firebase acepta la solicitud.

ACCESO ADMIN
------------
- Login usa la misma cuenta Firebase.
- role ADMIN en users/{uid} provoca navegacion a AdminDashboard.
- Registro siempre crea USER. El primer ADMIN se asigna manualmente en consola.
- requireAdmin vuelve a leer el perfil antes de consultas/acciones administrativas.
- Nunca se guarda contraseña administrativa en codigo.

DASHBOARD
---------
FirebaseAdminRepository.dashboard:
1. requireAdmin.
2. Lee users, offers, transactions, disputes.
3. Cuenta usuarios, ofertas ACTIVA, transacciones COMPLETADO y disputas PENDIENTE.
4. AdminDashboardScreen representa esos datos y accesos rapidos.

RESOLUCION DE DISPUTA
---------------------
resolveDispute es la funcion administrativa mas sensible:
1. Comprueba ADMIN, nota no vacia y disputa PENDIENTE.
2. Obtiene transaccion EN_DISPUTA, owner, recipient, moneda y retenido.
3. Lee balance retenido.
4. Decision liberar: resta retained al owner y suma available al recipient.
5. Decision devolver: resta retained y devuelve available al owner.
6. Actualiza transaccion a COMPLETADO o CANCELADO.
7. Disputa pasa a RESUELTA con decision, nota, resolvedBy y timestamp.
8. Crea movimiento y notifica buyer/seller.
9. Todo sucede en una transaccion Firestore.

FEEDBACK, REPORTES Y ALERTAS
----------------------------
- feedback consulta todos para ADMIN; respondFeedback actualiza respuesta/estado y crea
  notificacion al usuario.
- report selecciona coleccion segun tipo y filtra moneda/estado en el prototipo.
- AdminNotificationsScreen deriva alertas de disputas y feedback pendientes.
- No exporta Excel/PDF actualmente; muestra resultados dentro de la app.

REGLAS FIRESTORE
----------------
Funciones: signedIn, isOwner, isAdmin, transactionParticipant, disputeParticipant.
Ejemplos:
- users: propietario/admin; usuario no cambia role.
- transactions: participantes/admin.
- paymentData: dueño/admin.
- disputes: participantes/admin.
- notifications: destinatario/admin; destinatario solo cambia read/readAt.
- attachments: autenticado, JPEG y <=512000 bytes.
- regla final deniega cualquier coleccion no declarada.

LIMITACION IMPORTANTE
---------------------
El prototipo ejecuta movimientos desde el cliente para evitar backend de pago. Las reglas reducen
acceso, pero seguridad financiera de produccion necesita Cloud Functions/servidor confiable,
validacion real de Yape/banco, auditoria y secretos fuera de APK.

CONFIGURACION
-------------
- firebase.json apunta a firestore.rules.
- Reglas se publican en Firebase Console o firebase deploy --only firestore:rules.
- google-services.json conecta proyecto, queda en app/ y .gitignore.
- AndroidManifest.xml declara INTERNET y desactiva backup.
- app/build.gradle.kts declara Firebase/Compose/Navigation/Analytics.
- ExchangeRateRepository consume Frankfurter; si falla usa fallback.

PRUEBAS Y GIT
-------------
Comandos:
gradlew.bat assembleDebug
gradlew.bat testDebugUnitTest
gradlew.bat lintDebug

El repositorio no contiene datos de Firestore ni usuarios; Git solo versiona codigo/configuracion.
Los cambios se desarrollan en feature/firebase-integracion y se fusionan a master por Pull Request.

3. Responder feedback y revisar notificacion del usuario.

IDEA CENTRAL PARA EXPLICAR
--------------------------
El administrador no es una interfaz separada sin control: el rol se comprueba en datos y reglas.
Las decisiones que mueven dinero se ejecutan atomicamente y dejan estado, movimiento y avisos.
