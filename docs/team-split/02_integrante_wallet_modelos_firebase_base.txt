PROPOSITO DE ESTAS GUIAS
------------------------
Cada integrante recibe el flujo completo y una especialidad. Las rutas parten de
D:\MOVILES_PRO_2.

FLUJO GENERAL COMPLETO DE EXCHANGEPRO
-------------------------------------
1. MainActivity aplica el tema y abre ExchangeProNavGraph.
2. Login usa FirebaseAuthRepository; Firebase Authentication valida y users/{uid} define rol.
3. Registro crea perfil, wallet y balances PEN/USD/EUR mediante un batch.
4. Home combina perfil, saldos, ofertas y transacciones.
5. Wallet consulta balances y movimientos; recarga/retiro usan transacciones atomicas.
6. paymentData almacena Yape, Plin y banco del usuario.
7. Crear oferta retiene saldo; tomar oferta crea transaccion.
8. El comprobante se comprime y almacena en attachments.
9. Liberar fondos mueve saldo retenido y completa la operacion.
10. Una disputa congela la operacion; ADMIN decide el destino.
11. Notificaciones, feedback y ratings completan la interaccion.
12. firestore.rules limita acceso por uid, participante y rol.

-------------------
Pantalla -> Repository -> coleccion Firestore -> data class -> estado Compose -> pantalla.
domain/model define el lenguaje comun; data/repository implementa persistencia; presentation

MODELOS Y COLECCIONES
---------------------
- CurrencyCode evita monedas escritas de maneras distintas.
- Wallet contiene userId y lista de WalletBalance.
- WalletBalance separa available de retained. Retenido no puede retirarse mientras sostiene
  una oferta/transaccion.
- WalletMovement representa el historial.
- TopUpRequest y WithdrawalRequest llevan datos validados desde la UI al repositorio.
- FirebaseCollections centraliza users, wallets, offers, transactions, etc.; cambiar un nombre
  alli impacta todas las consultas.

LECTURA DE WALLET
-----------------
FirebaseWalletRepository.getWallet:
1. Obtiene uid de FirebaseAuth.
2. Lee wallets/{uid}/balances.
3. Convierte documentos a WalletBalance.
4. Ordena por moneda y devuelve Wallet.

getMovements consulta walletMovements por userId, transforma Timestamp y ordena en cliente.
WalletScreen llama refresh dentro de coroutine, actualiza wallet/movements/destinos y Compose
redibuja automaticamente.

RECARGA
-------
1. El usuario elige PEN/USD/EUR, monto, cuenta receptora de ExchangePro y referencia.
2. La referencia es el codigo que entrega Yape/Plin/banco; en el prototipo se registra pero no
   se valida contra una API bancaria.
3. topUp ejecuta una transaccion Firestore.
4. Lee el saldo actual, incrementa available, crea topUps y walletMovements.
5. Si una escritura falla, ninguna se confirma.
6. Es simulacion academica: en produccion un backend verificaria el pago antes de acreditar.

RETIRO
------
1. PaymentDataScreen guarda destinos del usuario en paymentData/{uid}.
2. availableDestinations convierte campos no vacios en opciones.
3. WithdrawDialog permite elegir moneda y destino.
4. withdraw lee saldo y paymentData dentro de una transaccion.
5. Verifica saldo suficiente, descuenta available y crea withdrawals + movimiento negativo.
6. El prototipo lo marca completado; produccion requeriria proveedor de pagos/backend.

SALDO TOTAL Y TASAS
-------------------
- ExchangeRateRepository consulta Frankfurter v2 (api.frankfurter.dev) mediante HttpURLConnection.
- La API devuelve moneda extranjera por PEN; el repositorio invierte la equivalencia.
- Compra/venta usan un margen porcentual de 0.8 %, valido tambien para monedas pequeñas como JPY.
- Si Internet falla, fallbackRates mantiene valores de demostracion.
- WalletHero convierte disponible + retenido a PEN usando buy y muestra "estimado".
- Los saldos originales nunca cambian por mostrar el total.

DATOS DE PAGO
-------------
- FirebasePaymentDataRepository.get/save/delete usa un documento por uid.
- La UI valida longitud de telefonos, cuenta y CCI segun banco.
- Yape/Plin del perfil no se suponen automaticamente: el usuario registra destinos en Datos
  de pago. Las cuentas mostradas al recargar son cuentas receptoras de ExchangePro.

IDEA CENTRAL PARA EXPLICAR
--------------------------
