INTEGRANTE 4 - CICLO DE TRANSACCION, IMAGENES, DISPUTAS Y COMUNICACION
=====================================================================

PROPOSITO DE ESTAS GUIAS
------------------------
Esta guia repite el contexto general y profundiza desde que existe una transaccion hasta su
finalizacion, disputa o calificacion.

FLUJO GENERAL COMPLETO DE EXCHANGEPRO
-------------------------------------
1. MainActivity abre NavGraph y Login.
2. Firebase autentica; Firestore define rol.
3. Registro crea usuario, wallet y balances.
4. Home reúne datos de varios repositorios.
5. Wallet/paymentData preparan fondos y medios.
6. Oferta retiene fondos; tomar oferta crea transaccion.
7. Pagador adjunta comprobante comprimido.
8. Contraparte libera fondos o abre disputa.
9. Completar mueve saldo; cancelar restaura oferta; disputa congela.
10. Ratings, notificaciones y feedback quedan en Firestore.
11. Admin resuelve pendientes.
12. Reglas protegen acceso.

COMO LEER EL CODIGO
-------------------
TransactionsScreen decide que botones mostrar segun status y rol dentro de la operacion.
FirebaseTransactionRepository aplica las transiciones. FirebaseAttachmentRepository gestiona
imagenes. Los repositorios de disputa/rating/notificacion/feedback completan el flujo.

MAQUINA DE ESTADOS
------------------
PENDIENTE_PAGO -> PAGADO -> COMPLETADO
PENDIENTE_PAGO -> CANCELADO
PENDIENTE_PAGO o PAGADO -> EN_DISPUTA -> COMPLETADO/CANCELADO por ADMIN
WALLET_INTERNA -> COMPLETADO directamente despues de validar e intercambiar ambos saldos

TransactionsScreen calcula:
- isBuyer/isSeller.
- canConfirmPayment: usuario que debe entregar la contraprestacion.
- canReleaseFunds: propietario del saldo retenido.
Esto evita mostrar la misma accion a ambas partes.

COMPROBANTE
-----------
1. ActivityResultContracts.GetContent abre selector del sistema con image/*.
2. ImageCompressor decodifica respetando orientacion, reduce dimensiones y calidad.
3. Repite compresion hasta <= 500 KB y produce CompressedImage.
4. uploadVoucher valida fundsRecipientId y PENDIENTE_PAGO.
5. En una transaccion crea attachments/{id}, guarda Blob JPEG y actualiza voucherAttachmentId,
   status PAGADO y paidAt.
6. Tambien notifica a fundsOwner para revisar.
7. getImage recupera bytes; TransactionsScreen decodifica Bitmap para vista previa.

POR QUE FIRESTORE Y NO STORAGE
------------------------------
Para el prototipo universitario se evito Cloud Storage con facturacion. Firestore limita cada
documento a 1 MiB, por eso se comprime a 500 KB y se guarda separado. Produccion debe usar
almacenamiento especializado.

COMPLETAR/CANCELAR
------------------
- complete solo lo ejecuta fundsOwner cuando status PAGADO.
- El receptor se acredita con FieldValue.increment, sin leer su balance privado.
- Resta heldAmount de retained del dueño y suma available del receptor.
- Crea dos movimientos, marca COMPLETADO y notifica.
- cancel solo antes del pago; devuelve monto/retencion a la oferta y marca CANCELADO.
- Todas son transacciones atomicas.

DISPUTAS
--------
1. DisputesScreen lista operaciones elegibles.
2. Usuario escribe motivo/descripcion y opcionalmente evidencia.
3. FirebaseDisputeRepository.create valida participante/estado.
4. Crea disputes/{id}, cambia transaccion a EN_DISPUTA y notifica a contraparte.
5. Evidencia usa uploadDisputeEvidence y arrayUnion de attachmentId.
6. El usuario no resuelve; lo hace ADMIN.

CALIFICACIONES
--------------
- Solo para COMPLETADO.
- Documento id transactionId_raterId evita duplicados.
- hasRated revisa existencia; submit valida score 1..5 y participante.
- Guarda ratedUserId y comentario. No se modifica libremente la reputacion de otro usuario
  desde el cliente por seguridad.

NOTIFICACIONES Y FEEDBACK
-------------------------
- FirebaseNotificationRepository consulta por userId y marca read/readAt.
- Operaciones principales crean avisos dentro de sus transacciones.
- FeedbackScreen crea RECOMENDACION o REPORTE_ERROR y lista respuestas del admin.

HOME Y PERFIL
-------------
- HomeScreen consulta perfil, wallet, ofertas y transacciones para un resumen.
- ProfileScreen usa FirebaseUserRepository y FirebaseAttachmentRepository para editar datos
  permitidos y foto comprimida.

DEMOSTRACION SUGERIDA
---------------------
1. Usuario B toma oferta; mostrar PENDIENTE_PAGO.
2. Subir captura; mostrar attachment y estado PAGADO.
3. Usuario A visualiza y libera; verificar balances/movimientos.
4. Calificar. En otra operacion abrir disputa y adjuntar evidencia.
5. Mostrar notificaciones de ambos usuarios.

IDEA CENTRAL PARA EXPLICAR
--------------------------
Los estados controlan que accion es legal. Las transacciones Firestore impiden que saldo,
movimiento y estado queden desincronizados; las imagenes son adjuntos referenciados.
